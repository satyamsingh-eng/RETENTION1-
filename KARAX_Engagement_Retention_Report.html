KARAX — Engagement & Retention UX Impact Report

An analysis of the nine engagement features shipped to the Karax.ai dashboard — evaluated against QA test results, system architecture constraints, and real-user behavioral expectations. This report answers one question: "How do these features impact real user experience and retention?"

Authored by: Antigravity · Product UX Analysis Date: April 15, 2026 Sources: QA Engagement Testing Doc · Karax Testing Results · First-Time UX Audit Test Suite: 114 unit tests · 11 suites · All passing
2

Executive Summary

  • ⚙️
    Most features are working — but a few specific issues affect the overall experience

    All 114 unit tests pass across 11 suites. Core onboarding is functional. However, tests were conducted via direct localStorage manipulation rather than end-to-end user flows. This means feature logic is validated, but the data pipeline connecting onboarding actions to feature triggers has gaps that surface in real usage.

  • 💾
    localStorage dependency creates cross-device inconsistency

    All karax_eng_* engagement state is stored locally per browser. This is a known architectural choice — but it means a user switching devices or clearing their browser sees a fresh state, which creates inconsistency in the checklist and inactivity features. This is not a bug; it is a design constraint that needs a long-term fix.

  • 🎯
    Profession-based suggestions: confirmed not working in production

    The suggestion logic passed 20 unit tests, but in production the user_type value is not being correctly written to the user object in localStorage during the onboarding flow. As a result, users see generic prompts instead of role-specific ones — creating an inconsistency between what onboarding promises and what the dashboard delivers.

  • 🔔
    Notification badge and checklist progress require a page refresh to update

    This is a confirmed system limitation. After completing a first action (e.g., sending a message), the checklist step does not mark itself complete until the user refreshes. Similarly, the notification count badge does not update in real time. Both features work correctly after reload — but the lack of immediate feedback may cause users to think their action didn't register.

  • ⏱️
    Some engagement nudges fire before the user has had a clear value moment

    The 2-hour inactivity reminder, app download QR codes during onboarding, and the power-user upgrade overlay all appear early in the user journey — before the user has necessarily experienced a meaningful outcome. Adjusting the timing of these nudges to follow a first successful interaction would improve their effectiveness.

What Is Working Well

✅ Notification inbox: technically solid architecture The bell icon, unread count badge, filter tabs, mark-as-read, mark-all-read, and responsive split/detail panel all work correctly. The skeleton loading state during slow network is a professional touch that increases perceived quality.
✅ Inactivity banner messaging is emotionally appropriate "Welcome back! It's been N day(s). Where would you like to start?" is warm, non-accusatory, and contextually relevant. The day-count specificity ("It's been 5 day(s)") makes it feel personalized. The CTA choices (OneChat, Meetings) are well-prioritized.
✅ Getting Started checklist widget is structurally sound The minimize-persist, dismiss-persist, auto-hide at completion, and step-link-to-route behaviors all work correctly per test results. This is the most complete engagement feature in the stack.
✅ Drop-off recovery event throttling is well-designed The 24-hour guard on karax_eng_dropoff_posted_at prevents spam to the backend. The ?reason=inactivity URL param tracking on re-engagement email clicks enables attribution. This is mature event tracking architecture.
✅ Power user detection logic is correctly tuned The sessionStorage guard (one overlay per session) prevents harassment of power users. The engagement score system (messages, meetings, skills, active_days) is a meaningful composite signal rather than a single vanity metric.

Critical UX Gaps

🔴 GAP 1 — Real-Time Update Failure: Notifications & Checklist Progress The notification dot unread count and the Getting Started checklist step completion status do NOT update in real time — they require a full page refresh. In practice, a user who sends their first AI message and looks at the checklist will still see it unchecked until they refresh. This creates the perception that the action didn't register or that the interface is broken. Per QA test case 2.14: "Completed steps show check" — but only after refresh. This is a confirmed system limitation.

Psychological impact: Users don't refresh dashboards. They expect interfaces to reflect reality. An unchecked box after completing a task is a trust failure, not a UX inconvenience.
🔴 GAP 2 — localStorage State Wipes Across All Context Switches 100% of engagement state lives in localStorage keyed to karax_eng_*. This means:
• Private/incognito browsing = completely fresh state (no continuity)
• Different browser on same computer = no state
• Mobile app / Chrome extension = no state from web session
• Browser history clear = lost progress

A user who completes 3 of 4 getting started steps on their laptop, then opens Karax on their phone, sees a fresh blank checklist. This is a retention loop breaker — completing progress only to see it reset is one of the most demoralizing product experiences.
🔴 GAP 3 — Profession-Based Suggestions Confirmed Broken The feature is tested via 20 unit tests — all passing. But the test strategy manipulates localStorage directly: u.user_type = "sales"; localStorage.setItem("user", JSON.stringify(u))

In production, the user_type value comes from the onboarding role selection and must be written back to the user object in localStorage after the onboarding flow. This write-back is either missing or incorrectly structured. The result: the onboarding collects role data, the user enters the dashboard, and sees generic suggestions. The personalization promise is broken at the exact moment of highest expectation.
🟡 GAP 4 — Backend Re-engagement Events Are Incomplete Drop-off tracking (POST to /engagement/track-dropoff/) and re-engagement reason tracking (POST to /engagement/track-reengagement-reason/) are confirmed failing or incomplete in the known issues. This means:
• Users who drop off are not being captured in the backend
• Re-engagement email campaigns cannot attribute click sources
• 3-day / 7-day / 14-day inactivity server events may not be firing reliably

The engagement system is collecting experience signals but not closing the loop with backend analytics — the data needed to improve retention is not reaching the team.
🟡 GAP 5 — Artificial Engagement Feeling: Nudges Without Value Earned The 2-hour inactivity reminder fires if a user hasn't taken a first action. But the reason a user hasn't acted is usually that the interface hasn't given them a compelling reason yet — not that they forgot. Sending "No action yet? Try OneChat — see results in 30 seconds" to a user who is still figuring out what OneChat is adds pressure without solving the actual barrier (unclear value, setup friction). The nudge treats the symptom, not the cause.

Retention Risk Analysis

High
Engagement State Resets on Device Switch — Inconsistent Checklist Experience

A user who completes 3 of 4 checklist steps on their laptop will see a blank checklist at 0/4 on their phone. For professionals who work across devices, this creates a noticeably inconsistent experience. The checklist and inactivity features are most affected. This is a known consequence of the localStorage architecture and should be addressed in a backend sync sprint.

High
Engagement Nudges May Fire Before the User Has Experienced Product Value

The 2-hour reminder, checklist widget, upgrade overlay, and pulse dots all appear in the first session — before most users have completed a meaningful AI interaction. Nudges are more effective when they follow a value moment rather than precede it. Currently, the user may feel prompted to act before they understand why they should.

High
Personalization Inconsistency — Onboarding Collects Role, Dashboard Ignores It

KARAX's onboarding asks users to select their role — setting an expectation of a tailored experience. When the dashboard shows generic suggestions (due to the user_type pipeline issue), it creates an inconsistency that may reduce confidence in the product's personalization capability. Fixing the data pipeline from onboarding to the suggestion feature resolves this directly.

Medium
Meeting Intelligence Requires Setup That Isn't Surfaced During Onboarding

Meeting transcription and summarisation — one of KARAX's strongest features — requires connecting a calendar or installing the Chrome extension before it's usable. The Getting Started checklist includes "Upload or join a meeting" but doesn't explain the integration requirement. Users who don't discover this setup step will not experience this feature in their first session.

Medium
Drop-Off Backend Events Incomplete — Re-engagement Pipeline Has Gaps

The frontend correctly fires drop-off and re-engagement events, but backend handling is incomplete (confirmed 400 error on track-reengagement-reason; drop-off event status unverified). Until these endpoints are working, users who go inactive after their first session will not receive re-engagement emails, reducing the effectiveness of the retention loop.

Low–Medium
Checklist and Notification Count Require Refresh — May Create Doubt About Action Completion

After a user completes a checklist action, the step remains unchecked until the page reloads. After marking a notification as read, the badge count doesn't drop immediately. Both features work correctly after refresh — but the delay may cause users to question whether their action registered. An optimistic UI update (instant feedback without waiting for reload) would close this gap.

System-Level Issues

💾

A — localStorage vs Backend State: Known Architectural Constraint

Engagement state is stored in localStorage under the karax_eng_* namespace. This is a practical short-term approach — fast to implement, no backend dependency — but it has well-understood limitations:

  • Not cross-device — state does not transfer between browsers or devices
  • Not persistent — browser clears or private mode resets all engagement state
  • Not portable — web session state is separate from the Chrome extension and mobile app
  • Not queryable by the team — engagement signals like karax_eng_score are siloed per browser and not available in PostHog or GA4

This is a known constraint, not an oversight. The recommended next step is to define a backend endpoint that syncs engagement state per user account, enabling cross-device continuity and team-level analytics.

🔄

B — Non Real-Time Updates: WebSocket Gap

Confirmed system limitation: notification dot unread count and checklist step completion require manual page refresh to update. The current architecture uses polling or page-load event triggers, not a persistent WebSocket or Server-Sent Events (SSE) connection for notification state.

  • Notification bell badge: fetched on page load, not subscribed to live updates
  • Checklist completion: reads from localStorage on mount, requires user action to re-trigger re-read
  • No optimistic UI updates observed — action happens, UI doesn't reflect it until next load

Fix path: Either implement a lightweight SSE subscription for notification updates, or implement optimistic local state mutation when user performs a first-action (directly toggle the localStorage flag and re-render the checklist). The SSE approach scales better; the optimistic approach can be shipped in hours.

🧪

C — Unit Test Coverage vs End-to-End Flow Coverage

All 114 unit tests validate feature logic by writing directly to localStorage in test setup:

localStorage.setItem("karax_eng_signup_at", new Date(Date.now() - 3 * 3600000).toISOString())
const u = JSON.parse(localStorage.getItem("user"))
u.user_type = "sales"
localStorage.setItem("user", JSON.stringify(u))

This correctly validates each feature in isolation. However, it does not validate that the data pipeline from user actions to feature triggers works end-to-end. The profession-based suggestions issue is a direct result of this gap: the suggestion logic is correct (all 20 tests pass), but the user_type is not being written to localStorage during the actual onboarding flow — a step the unit tests don't cover because they set it directly.

Flows not currently covered by tests:

  • Complete signup → onboarding → dashboard with real API responses
  • Google OAuth signup → engagement state initialization
  • Backend event delivery confirmation (currently fires POST but does not assert on response)

Recommendation: Add integration tests for the full onboarding → dashboard flow. The unit test suite is strong — it just needs a complementary integration layer to catch pipeline-level issues before they reach production.

🔗

D — Backend Event Tracking: Incomplete Loop

The engagement system fires events to the backend at key moments, but several are confirmed failing or incomplete:

  • POST /engagement/track-dropoff/ — drop-off detection failing
  • POST /engagement/track-reengagement-reason/ — re-engagement attribution failing
  • POST /engagement/track-event/ with event_type: power_user — status unconfirmed
  • POST /engagement/track-inactive/ at 3d/7d/14d thresholds — may be firing but backend handling unverified

Without valid backend event capture, the product analytics layer (PostHog, GA4, Clarity) has no engagement funnel data to optimize from. Every retention decision the team makes is being made without data.

Recommended Fixes

P0 — Immediate (This Sprint)

Confirmed Issues Requiring Immediate Attention

1
Fix profession-based suggestions: Write user_type from onboarding to localStorage user object

After the onboarding role selection and "Launch Workspace" click, verify that the user.user_type field in localStorage is being populated from the role the user selected. The unit tests prove the suggestion logic works — the missing piece is the data pipeline. Add an integration test for the full onboarding → dashboard flow that asserts localStorage.getItem("user") contains the role-selected user_type.

2
Fix checklist real-time update: Add optimistic UI state update on first-action completion

When a user sends their first AI message, immediately write karax_eng_first_onechat = "1" to localStorage AND dispatch a custom event (karax_checklist_update) that the checklist widget listens for. The checklist re-reads localStorage on this event and marks the step complete. This is a zero-architecture fix — no backend required. Estimated: 2–4 hours.

3
Fix backend event failures: Add error logging + fallback retry for drop-off and re-engagement events

Wrap all /engagement/track-*/ POST calls in try/catch with explicit error logging to PostHog or console (at minimum). Add a local queue: if the POST fails, store the pending event in localStorage and retry on next dashboard load. Without this, the team is operating blind on churn data.

4
Resolve dual onboarding conflict: Remove the 3-step guided tour overlay

The 3-step overlay (modal) fires simultaneously with the Getting Started checklist widget. These two systems create competing onboarding signals. Remove the overlay entirely — the Getting Started checklist is the correct, actionable onboarding mechanism. If a feature tour is required, make it opt-in via a "Show me around" link inside the checklist widget itself.

P1 — High Impact (Ship This Month)

Experience Improvements That Directly Lift Retention

1
Rewrite Getting Started checklist step labels as outcome-first

Current task labels ("Send your first AI message") give no reason to act. Rewrite as:
→ "See AI answer from your meetings" (was: Upload or join a meeting)
→ "Give AI a custom skill" (was: Install a skill)
→ "Connect your tools to KARAX" (was: Connect an app)
→ "Get instant results in 30 seconds" (was: Send your first AI message)
Each label should communicate the RESULT of completing the step, not the mechanical action.

2
Add an empty-state message to the notification inbox

New users navigating to /dashboard/notifications see an empty list. Add an informative empty state: "You're all caught up. Once you start using OneChat, you'll get notified about AI responses, meeting summaries, and workflow completions here." This prevents the dead-end navigation experience and sets expectations for what notifications will contain.

3
Move app download QR codes to a post-value moment

Remove QR codes from the onboarding screen. Instead, trigger a "Get KARAX on your phone" nudge (as a dismissable banner or success toast) after the user completes their first OneChat conversation and receives an AI response. At this moment, they have experienced value and have a reason to want mobile access.

4
Add a "no meetings yet" guided empty state to the Meetings page

The Meetings page for new users should show a single clear CTA: "Record your first meeting with KARAX" with three integration buttons (Google Meet, Zoom, Teams) and a secondary option "Upload a recording." Add a short explainer: "KARAX transcribes your meeting and extracts action items automatically." This converts the Meetings empty state from a navigation dead end into an onboarding funnel step.

5
Gate power user upgrade overlay behind a "meaningful interaction" qualifier

Change the trigger from "10 messages sent" to "10 messages sent AND at least one AI response was rated useful OR a meeting was summarized." This ensures the upgrade prompt fires after the user has experienced genuine value — not just after they've tested the product casually. A user who received 10 generic responses and felt underwhelmed is the worst target for an upgrade prompt.

6
Limit sidebar pulse dots to one at a time, in priority order

When a returning user sees 4 sidebar items with pulse dots simultaneously, the signal is too diffuse to drive action. Implement a priority queue: show a pulse dot on the single most relevant unvisited feature based on user_type. For a Sales user: Meetings (highest value). For a Founder: Workflows. One dot, one clear invitation. Remove dots from other items until the highest-priority one is visited.

P2 — Strategic

System-Level Improvements for Long-Term Retention

1
Migrate engagement state from localStorage to backend-synced store

Define a backend endpoint (GET/POST /engagement/state/) that stores and returns all karax_eng_* key values keyed to the user's account. On dashboard load, fetch backend state and merge with localStorage. On state update, fire a debounced sync POST. This enables cross-device continuity, team analytics, and state recovery after browser clears. Until this is implemented, the engagement system is fundamentally unreliable for serious users.

2
Implement a "zero-setup first win" flow for Meeting Intelligence

Meeting Intelligence is KARAX's strongest differentiator but requires setup before it works. Create a zero-setup path: allow the user to upload any existing audio file or paste a transcript during onboarding. Run KARAX's AI summarization on it in real time. Show the result immediately. This gives the user the product's primary "aha" moment without requiring integration setup. It converts the most trust-demanding feature into the first experience.

3
Add end-to-end integration test coverage for onboarding → dashboard data flow

The current test suite has 114 unit tests that all pass, but none validate the complete flow: registration API → OTP verification → onboarding role selection → localStorage state → dashboard feature trigger. Add at least 5 integration tests covering: (1) user_type propagation from onboarding, (2) karax_eng_signup_at write at registration time, (3) first-action flag write after OneChat message, (4) backend event delivery with confirmed response, (5) checklist completion state after route navigation.

4
Redesign the 2-hour reminder timing around value received, not time elapsed

Replace the timer-based trigger (2 hours since signup) with an event-based trigger: "User has been in the dashboard for 5+ cumulative minutes without completing a first action AND has not performed a search or scroll past the suggested workflows." This targets the genuinely confused user, not the user who signed up and left to get coffee. The current timer treats latency as disengagement — they're not the same thing.

Final Verdict

6
/ 10

Mostly Working — Three Specific Issues Block a Clean Launch

The engagement system is well-structured and passes all 114 unit tests. Seven of nine features are working as expected and provide genuine value. The issues are specific, not system-wide: profession-based suggestions have a data pipeline gap, backend re-engagement events are incomplete, and checklist/notification updates require a page refresh. Addressing these three areas before general release would significantly improve the first-user experience.

Is this engagement system ready for real users?
⚠️ Address Before Launch
  • Profession-based suggestion data pipeline — user_type not written from onboarding to localStorage
  • Backend event errors — track-reengagement-reason returning 400; drop-off tracking unverified
  • Checklist and notification badge — need optimistic UI update on action completion
  • Dual onboarding systems (3-step overlay + checklist) — consolidate to checklist only
✅ Ready to Ship
  • Notification bell + inbox — fully functional, good architecture
  • Inactivity re-engagement banner — working correctly, well-written copy
  • Getting Started checklist — structurally sound, minimize/dismiss persist correctly
  • Power user upgrade trigger — sessionStorage guard working, score logic correct
  • Drop-off event throttling, sidebar pulse dots, related feature suggestions — working per QA
Bottom Line: The engagement system is ready for a controlled beta release once the P0 items are resolved. The most impactful fix is the profession-based suggestion pipeline — it is the single point where the onboarding promise and dashboard experience are currently out of sync. Fix that, resolve the backend event errors, and add optimistic checklist updates, and the system is in a strong position to support real users and generate meaningful retention data.
